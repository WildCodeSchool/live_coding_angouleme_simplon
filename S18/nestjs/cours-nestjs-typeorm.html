Qu'est-ce que NestJS ?

NestJS est un framework Node.js progressif pour construire des applications côté serveur efficaces, fiables et scalables. Écrit en TypeScript, il combine POO, Programmation Fonctionnelle et Programmation Réactive Fonctionnelle.

💡 Philosophie

NestJS résout un problème fondamental : Node.js offre une liberté totale d'architecture, mais cette liberté mène souvent à un code désorganisé. NestJS impose une structure claire et opinionée inspirée d'Angular, tout en restant compatible avec Express et Fastify en sous-couche.

Pourquoi NestJS ?

🏗️

Architecture structurée

Organisation claire via des modules, contrôleurs et services. Idéal pour les grandes équipes et les projets durables.

🔷

TypeScript natif

Typage statique, autocomplétion, détection d'erreurs à la compilation. Développement plus sûr et productif.

⚡

Performance Express/Fastify

Basé sur Express par défaut (ou Fastify), NestJS hérite de leurs performances tout en ajoutant une couche d'abstraction élégante.

🔌

Injection de dépendances

Système d'IoC (Inversion of Control) intégré. Couplage faible et testabilité maximale entre les composants.

🧩

Ecosystème riche

CLI puissante, support natif de WebSockets, GraphQL, microservices, gRPC, CQRS, et bien plus encore.

🧪

Testabilité

Architecture pensée pour les tests unitaires et d'intégration, compatible Jest out-of-the-box.

02

Architecture & Approche

NestJS adopte une architecture modulaire et en couches, fortement inspirée d'Angular. Chaque fonctionnalité est encapsulée dans un Module, qui regroupe des Controllers, des Services et d'autres providers.

Architecture en couches (Layered Architecture)

🎮

Couche Présentation

Controllers — Reçoit les requêtes HTTP, délègue la logique aux services, retourne les réponses.

⚙️

Couche Métier

Services / Providers — Contient toute la logique métier de l'application. Réutilisable et testable.

🗄️

Couche Données

Repositories / TypeORM — Gère l'accès à la base de données. Abstraction complète des requêtes SQL.

Le Module : brique fondamentale

Un module NestJS est une classe décorée avec @Module(). Chaque application a au minimum un AppModule racine. Les modules s'organisent par domaine fonctionnel et gèrent leurs propres dépendances.

app.module.ts
TypeScript
import { Module } from '@nestjs/common';
import { UsersModule } from './users/users.module';
import { ProductsModule } from './products/products.module';

@Module({
  imports: [UsersModule, ProductsModule],  // Modules importés
  controllers: [],                           // Controllers du module racine
  providers: [],                              // Services / Providers
  exports: [],                                // Exposé à d'autres modules
})
export class AppModule {}

Principes de conception

SOLID appliqué à NestJS

  • S — Un service = une responsabilité précise
  • O — Extension via modules sans modifier l'existant
  • L — Substitution via l'injection de dépendances
  • I — Interfaces granulaires pour les providers
  • D — Dépendance aux abstractions, pas aux implémentations

DDD-friendly

  • Organisation par domaine métier (users, orders, products…)
  • Bounded contexts représentés par des modules
  • Entités et value objects via TypeORM
  • Repository pattern natif
  • Events via EventEmitter ou module CQRS
✅ Bonne pratique

Organisez vos modules par domaine fonctionnel, pas par type technique. Préférez un UsersModule contenant controller + service + entity à des répertoires controllers/ et services/ séparés à la racine.

03

Concepts Clés

NestJS repose sur un ensemble de blocs de construction qui structurent le traitement des requêtes de façon prévisible et extensible.

Decorator

Décorateurs

Annotations TypeScript (syntaxe @NomDecorator) enrichissant les classes, méthodes ou paramètres avec des métadonnées. Au cœur de NestJS : @Controller(), @Get(), @Injectable(), @Body(), @Param()...

Pipe

Pipes

Transforment et valident les données d'entrée avant qu'elles atteignent le handler. ValidationPipe valide les DTOs, ParseIntPipe convertit strings → entiers. Lèvent automatiquement une erreur 400 si invalide.

Middleware

Middleware

S'exécutent avant le routing. Identiques aux middlewares Express : accèdent aux objets req, res et next(). Utilisés pour les logs, la compression, CORS, l'authentification basique.

Service / Provider

Services & Providers

Classes décorées @Injectable() injectées via le système IoC. Contiennent la logique métier. Un provider peut être un service, un repository, une factory, une valeur constante, etc.

Guard

Guards

Déterminent si une requête doit être traitée. Implémentent CanActivate. Utilisés pour l'authentification JWT et l'autorisation RBAC. Retournent true (autorisé) ou false (403 Forbidden).

Interceptor

Interceptors

Inspirés de l'AOP (Aspect-Oriented Programming). S'exécutent avant ET après le handler. Permettent : transformation des réponses, mise en cache, logging des performances, gestion des timeouts.

Module

Modules

Organisent l'application en unités fonctionnelles. Chaque module encapsule son contexte (controllers, providers). Le système d'imports/exports gère les dépendances inter-modules proprement.

Controller

Controllers

Gèrent les routes HTTP entrantes. Chaque méthode correspond à un endpoint : @Get(), @Post(), @Put(), @Delete(), @Patch(). Délèguent toute la logique aux services.

Exemple : Controller complet

users.controller.ts
TypeScript
import { Controller, Get, Post, Put, Delete,
         Body, Param, ParseIntPipe, UseGuards,
         ValidationPipe, HttpCode, HttpStatus } from '@nestjs/common';
import { UsersService } from './users.service';
import { CreateUserDto } from './dto/create-user.dto';
import { JwtAuthGuard } from '../auth/jwt-auth.guard';

@Controller('users')           // Préfixe de route : /users
export class UsersController {

  // Injection du service via le constructeur
  constructor(private readonly usersService: UsersService) {}

  @Get()                       // GET /users
  findAll() {
    return this.usersService.findAll();
  }

  @Get(':id')                  // GET /users/:id
  findOne(@Param('id', ParseIntPipe) id: number) {
    return this.usersService.findOne(id);
  }

  @Post()                      // POST /users
  @HttpCode(HttpStatus.CREATED)
  create(@Body(ValidationPipe) dto: CreateUserDto) {
    return this.usersService.create(dto);
  }

  @Put(':id')                  // PUT /users/:id — Protégé par JWT
  @UseGuards(JwtAuthGuard)
  update(@Param('id', ParseIntPipe) id: number, @Body() dto: any) {
    return this.usersService.update(id, dto);
  }

  @Delete(':id')               // DELETE /users/:id
  @HttpCode(HttpStatus.NO_CONTENT)
  remove(@Param('id', ParseIntPipe) id: number) {
    return this.usersService.remove(id);
  }
}

import { IsEmail, IsString, MinLength, IsOptional, IsEnum } from 'class-validator';

export class CreateUserDto {
  @IsString()
  @MinLength(2)
  name: string;

  @IsEmail()
  email: string;

  @IsString()
  @MinLength(8)
  password: string;

  @IsOptional()
  @IsEnum(['admin', 'user'])
  role?: string;
}

Cycle de vie d'une Requête HTTP

Lorsqu'une requête HTTP arrive dans NestJS, elle traverse plusieurs couches dans un ordre précis et déterministe. Comprendre ce pipeline est essentiel pour bien placer votre logique.

01
Requête
Client HTTP envoie la requête
02
Middleware
Traitement avant routing (logs, CORS...)
03
Guard
Autorisation & authentification
04
Interceptor
Logique pré-traitement (cache, metrics)
05
Pipe
Validation & transformation des inputs
06
Controller
Handler de route appelé
07
Service
Logique métier exécutée
08
Interceptor
Post-traitement de la réponse
09
Exception Filter
Gestion des erreurs si exception
10
Réponse
JSON envoyé au client

Détail de chaque étape

#ComposantRôleScope possible
01MiddlewareAccès à req/res avant routing, next() obligatoireGlobal / Module / Route
02GuardRetourne true/false, bloque si non autorisé → 403Global / Controller / Route
03Interceptor (pre)Logique avant handler : timing, logging, transformation reqGlobal / Controller / Route
04PipeValide et transforme les arguments du handlerGlobal / Controller / Route / Param
05Controller HandlerMéthode du controller appelée, délègue au serviceRoute
06Interceptor (post)Transforme la réponse, mise en cache, log de performanceGlobal / Controller / Route
07Exception FilterCapture les exceptions non gérées, formate la réponse d'erreurGlobal / Controller / Route
⚠️ Ordre d'exécution

L'ordre est toujours : Global → Controller → Route. Les éléments globaux sont enregistrés via app.useGlobalGuards(), app.useGlobalPipes() etc. dans le main.ts.

Exemple : Exception Filter personnalisé

http-exception.filter.ts
TypeScript
import { ExceptionFilter, Catch, ArgumentsHost, HttpException } from '@nestjs/common';

@Catch(HttpException)
export class HttpExceptionFilter implements ExceptionFilter {
  catch(exception: HttpException, host: ArgumentsHost) {
    const ctx = host.switchToHttp();
    const response = ctx.getResponse();
    const status = exception.getStatus();

    response.status(status).json({
      statusCode: status,
      timestamp: new Date().toISOString(),
      message: exception.message,
    });
  }
}

Qu'est-ce que TypeORM ?

TypeORM est un ORM (Object-Relational Mapper) open-source pour TypeScript et JavaScript, inspiré de Hibernate (Java), Doctrine (PHP) et Entity Framework (.NET). Il fait le pont entre vos classes TypeScript et vos tables SQL.

Quel problème résout un ORM ?

🤔 Le problème : l'impedance mismatch

Les bases de données relationnelles stockent des données en tables avec des lignes et colonnes. Les applications orientées objet travaillent avec des objets, classes et relations. Cet écart fondamental s'appelle "impedance mismatch". L'ORM résout ce problème en servant de couche de traduction automatique.

Sans ORM — SQL brut
// ❌ Problèmes : SQL brut, mapping
// manuel, pas de typage, injection SQL
const result = await db.query(
  `SELECT u.*, p.title FROM users u
   LEFT JOIN posts p ON p.author_id = u.id
   WHERE u.id = ? AND u.active = 1`,
  [userId]
);

// Mapping manuel fastidieux...
const user = {
  id: result[0].id,
  name: result[0].name,
  posts: result.map(r => ({ title: r.title }))
};
Avec TypeORM
// ✅ Élégant : typé, lisible, sécurisé

const user = await this.userRepo.findOne({
  where: { id: userId, isActive: true },
  relations: ['posts'],
});

// ✓ Résultat typé en User automatiquement
// ✓ Pas d'injection SQL possible
// ✓ Code réutilisable et testable
// ✓ Relations chargées proprement

Pourquoi utiliser un ORM ?

🔒

Sécurité contre l'injection SQL

Protection automatique via les requêtes paramétrées. Fini les concaténations de chaînes dangereuses dans les requêtes SQL.

⚡

Productivité accrue

CRUD automatique, migrations gérées, relations déclarées en TypeScript. Moins de code boilerplate, plus de valeur ajoutée.

🔷

Typage TypeScript fort

Entités = classes TypeScript typées. L'IDE fournit l'autocomplétion et détecte les erreurs de propriétés à la compilation.

🗄️

Portabilité multi-SGBD

Changez de base (MySQL → PostgreSQL) avec un minimum de modifications. TypeORM s'adapte automatiquement à la syntaxe SQL.

📦

Migrations versionnées

Versionnez les changements de schéma. Synchronisez les environnements de façon reproductible et sûre en production.

🔗

Relations simplifiées

OneToOne, OneToMany, ManyToMany déclarés proprement avec lazy/eager loading. Fini les JOINs SQL complexes écrits à la main.

Bases de données supportées

TypeORM supporte : MySQL, PostgreSQL, SQLite, MariaDB, Oracle, MS SQL Server, MongoDB (expérimental), CockroachDB, et SAP Hana.

Les Entités TypeORM

Une entité est une classe TypeScript décorée @Entity() correspondant à une table SQL. Chaque propriété décorée correspond à une colonne.

user.entity.ts
TypeScript
import { Entity, PrimaryGeneratedColumn, Column,
         CreateDateColumn, UpdateDateColumn, OneToMany } from 'typeorm';
import { Post } from '../posts/entities/post.entity';

@Entity('users')                       // Nom de la table SQL
export class User {

  @PrimaryGeneratedColumn()            // INT AUTO_INCREMENT
  id: number;

  @Column({ length: 100 })
  name: string;

  @Column({ unique: true })
  email: string;

  @Column({ select: false })           // Exclue des SELECT par défaut
  password: string;

  @Column({ default: true })
  isActive: boolean;

  @Column({ type: 'enum', enum: ['admin', 'user'], default: 'user' })
  role: string;

  @OneToMany(() => Post, post => post.author)
  posts: Post[];                        // Relation : un user a plusieurs posts

  @CreateDateColumn()                  // Rempli auto à la création
  createdAt: Date;

  @UpdateDateColumn()                  // Mis à jour automatiquement
  updatedAt: Date;
}

Les Relations TypeORM

RelationDécorateursExemple concretSQL généré
Un à un@OneToOneUser ↔ ProfileForeign key unique sur une table
Un à plusieurs@OneToMany / @ManyToOneUser → PostsForeign key sur la table "Many"
Plusieurs à plusieurs@ManyToMany + @JoinTableUsers ↔ RolesTable de jointure intermédiaire

Le QueryBuilder pour les requêtes complexes

users.service.ts
TypeScript
// Requête complexe avec jointures, conditions et pagination
const [users, total] = await this.userRepo
  .createQueryBuilder('user')
  .leftJoinAndSelect('user.posts', 'post')
  .where('user.isActive = :active', { active: true })
  .andWhere('user.role = :role', { role: 'admin' })
  .andWhere('post.published = :pub', { pub: true })
  .orderBy('user.createdAt', 'DESC')
  .take(10)                          // LIMIT
  .skip(0)                           // OFFSET
  .getManyAndCount();                 // Retourne [data, total]

Étape 4 — Entité Post avec relation

post.entity.ts
TypeScript
import { Entity, PrimaryGeneratedColumn, Column,
         ManyToOne, CreateDateColumn, JoinColumn } from 'typeorm';
import { User } from '../../users/entities/user.entity';

@Entity('posts')
export class Post {

  @PrimaryGeneratedColumn()
  id: number;

  @Column({ length: 255 })
  title: string;

  @Column({ type: 'text' })
  content: string;

  @Column({ default: false })
  published: boolean;

  // Plusieurs posts appartiennent à un auteur (User)
  @ManyToOne(() => User, user => user.posts, { eager: true })
  @JoinColumn({ name: 'author_id' })
  author: User;

  @CreateDateColumn()
  createdAt: Date;
}

posts.service.ts
TypeScript
import { Injectable, NotFoundException } from '@nestjs/common';
import { InjectRepository } from '@nestjs/typeorm';
import { Repository } from 'typeorm';
import { Post } from './entities/post.entity';

@Injectable()
export class PostsService {
  constructor(
    @InjectRepository(Post)
    private readonly postsRepo: Repository<Post>,
  ) {}

  findAll(page = 1, limit = 10) {
    return this.postsRepo.findAndCount({
      where: { published: true },
      order: { createdAt: 'DESC' },
      take: limit,
      skip: (page - 1) * limit,
    });
  }

  async findOne(id: number) {
    const post = await this.postsRepo.findOneBy({ id });
    if (!post) throw new NotFoundException(`Post #${id} introuvable`);
    return post;
  }

  async create(dto: Partial<Post>, authorId: number) {
    const post = this.postsRepo.create({ ...dto, author: { id: authorId } });
    return await this.postsRepo.save(post);
  }

  async update(id: number, dto: Partial<Post>) {
    await this.findOne(id);
    await this.postsRepo.update(id, dto);
    return this.findOne(id);
  }

  async remove(id: number) {
    await this.findOne(id);
    await this.postsRepo.delete(id);
    return { message: `Post #${id} supprimé` };
  }
}

Étape 6 — main.ts (point d'entrée)

main.ts
TypeScript
import { NestFactory } from '@nestjs/core';
import { ValidationPipe } from '@nestjs/common';
import { AppModule } from './app.module';

async function bootstrap() {
  const app = await NestFactory.create(AppModule);

  // Validation globale avec class-validator
  app.useGlobalPipes(new ValidationPipe({
    whitelist: true,             // Supprime les props non décorées
    forbidNonWhitelisted: true, // Erreur si prop inconnue
    transform: true,             // Auto-transformation des types
  }));

  app.setGlobalPrefix('api/v1');  // Préfixe global
  app.enableCors({ origin: 'http://localhost:3000' });

  await app.listen(3001);
  console.log('🚀 API running on http://localhost:3001/api/v1');
}
bootstrap();
