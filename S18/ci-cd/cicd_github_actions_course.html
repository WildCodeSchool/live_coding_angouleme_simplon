🔄 Qu'est-ce que CI/CD ?

CI - Intégration Continue (Continuous Integration)

Pratique consistant à intégrer fréquemment les modifications de code dans un référentiel partagé, où des tests automatisés vérifient chaque intégration.

CD - Déploiement Continu (Continuous Deployment/Delivery)

Extension de CI qui automatise le déploiement des applications après validation des tests.

✅ Avantages du CI/CD

  • Détection précoce des erreurs - Les bugs sont identifiés rapidement
  • Réduction des risques - Déploiements plus petits et fréquents
  • Livraison plus rapide - Automatisation des processus manuels
  • Amélioration de la qualité - Tests automatisés systématiques
  • Collaboration renforcée - Intégration fluide des équipes
  • Retour rapide - Feedback immédiat sur les modifications
Important : Le CI/CD n'est pas seulement un outil, c'est une culture et une pratique qui nécessite l'engagement de toute l'équipe.

🚀 Introduction à GitHub Actions

GitHub Actions est une plateforme d'automatisation CI/CD intégrée directement dans GitHub.

Caractéristiques principales :

  • Intégration native avec les repositories GitHub
  • Workflows basés sur des événements
  • Marketplace d'actions réutilisables
  • Runners hébergés ou auto-hébergés
  • Support multi-plateformes (Linux, Windows, macOS)
Avantage : Pas besoin de service externe - tout est intégré dans GitHub !

🧩 Concepts de Base

Workflow (Flux de travail)

Processus automatisé défini dans un fichier YAML qui s'exécute lors d'événements spécifiques.

Events (Événements)

Déclencheurs qui lancent l'exécution d'un workflow (push, pull request, schedule, etc.).

Jobs (Tâches)

Ensemble d'étapes qui s'exécutent sur le même runner.

Steps (Étapes)

Tâches individuelles qui exécutent des commandes ou des actions.

Actions

Applications personnalisées qui effectuent des tâches complexes fréquemment répétées.

📋 Structure d'un Workflow


                name: Mon Premier Workflow
                on: [push, pull_request]

                jobs:
                build:
                runs-on: ubuntu-latest
                steps:
                - name: Checkout du code
                uses: actions/checkout@v3

                - name: Setup Node.js
                uses: actions/setup-node@v3
                with:
                node-version: '18'

                - name: Installation des dépendances
                run: npm install

                - name: Exécution des tests
                run: npm test

Les workflows sont stockés dans .github/workflows/ à la racine du repository.

⚡ Événements (Events)

Événements principaux :

  • push - Déclenchement lors d'un push
  • pull_request - Ouverture/mise à jour d'une PR
  • schedule - Exécution programmée (cron)
  • workflow_dispatch - Déclenchement manuel
  • release - Création d'une release

                # Exemples de configuration d'événements
                on:
                    push:
                      branches: [ main, develop ]
                    pull_request:
                      branches: [ main ]
                    schedule:
                       - cron: '0 2 * * *' # Tous les jours à 2h
                    workflow_dispatch: # Déclenchement manuel

⚙️ Jobs - Tâches

Les jobs s'exécutent en parallèle par défaut, mais peuvent être configurés pour s'exécuter séquentiellement.


                jobs:
                test:
                runs-on: ubuntu-latest
                steps:
                - uses: actions/checkout@v3
                - run: npm test

                build:
                needs: test # Attend que 'test' soit terminé
                runs-on: ubuntu-latest
                steps:
                - uses: actions/checkout@v3
                - run: npm run build

                deploy:
                needs: [test, build] # Attend test ET build
                runs-on: ubuntu-latest
                if: github.ref == 'refs/heads/main'
                steps:
                - run: echo "Déploiement en production"

🔧 Steps et Actions

Types d'étapes :

  • run - Exécute des commandes shell
  • uses - Utilise une action pré-construite

                steps:
                # Utilisation d'une action
                - name: Checkout du repository
                uses: actions/checkout@v3

                # Exécution d'une commande
                - name: Afficher les fichiers
                run: ls -la

                # Commande avec variables d'environnement
                - name: Build avec variables
                run: |
                echo "Building for ${{ github.ref }}"
                npm run build
                env:
                NODE_ENV: production

🏃 Runners

Les runners sont les machines qui exécutent vos workflows.

Types de runners :

  • GitHub-hosted - Gérés par GitHub
  • Self-hosted - Gérés par vous

Runners hébergés par GitHub :


                runs-on: ubuntu-latest # Ubuntu Linux
                runs-on: windows-latest # Windows Server
                runs-on: macos-latest # macOS
Astuce : Les runners GitHub-hosted sont gratuits pour les repositories publics avec des limites de temps d'exécution.

🔐 Variables et Secrets

Variables d'environnement :


                env:
                NODE_ENV: production
                API_URL: https://api.example.com

                steps:
                - name: Utilisation des variables
                run: echo "Environment: $NODE_ENV"

Secrets :

Stockage sécurisé d'informations sensibles (API keys, mots de passe, etc.)


                steps:
                - name: Déploiement avec secret
                run: |
                echo "Déploiement vers ${{ secrets.PROD_SERVER }}"
                # Utilisation du secret pour l'authentification
                env:
                API_KEY: ${{ secrets.API_KEY }}
Sécurité : Ne jamais exposer les secrets dans les logs ou les écrire en dur dans le code !

📝 Exemple Complet - Application Node.js


                name: CI/CD Pipeline

                on:
                push:
                branches: [ main, develop ]
                pull_request:
                branches: [ main ]

                jobs:
                test:
                runs-on: ubuntu-latest
                strategy:
                matrix:
                node-version: [16, 18, 20]

                steps:
                - uses: actions/checkout@v3

                - name: Setup Node.js ${{ matrix.node-version }}
                uses: actions/setup-node@v3
                with:
                node-version: ${{ matrix.node-version }}
                cache: 'npm'

                - name: Install dependencies
                run: npm ci

                - name: Run tests
                run: npm test

                - name: Upload coverage reports
                uses: codecov/codecov-action@v3

🎯 Bonnes Pratiques

  • Noms explicites - Utilisez des noms clairs pour workflows et jobs
  • Tests d'abord - Toujours tester avant de déployer
  • Matrice de tests - Testez sur plusieurs versions/plateformes
  • Gestion des secrets - Utilisez les secrets GitHub pour les données sensibles
  • Optimisation - Utilisez le cache pour accélérer les builds
  • Conditionnels - Utilisez des conditions pour contrôler l'exécution
  • Notification - Configurez des notifications pour les échecs
Conseil : Commencez simple et ajoutez progressivement de la complexité selon vos besoins.
